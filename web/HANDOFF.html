前端 → 后端联调交接说明 · 2026-05-17

这是一个给网剧团队创作网剧的工具,我负责前端,已经把 6 个页面都做完(工作台 / 编辑器 / 角色库 / 生成结果 / 使用记录 / 登录)。当前所有数据走前端本地 mock,需要后端实现真实接口后切换联调。

1. 交付内容 / 必读文件

文件作用
docs/api-contract.md 核心契约。所有接口的请求/响应字段、数据模型、错误码、鉴权细节都在这里。以此为准
web/src/types/*.ts TypeScript 类型定义,与契约一一对应。需要确认某个字段是否可空、类型是 string 还是 number,看这里最直观
web/src/api/_mock.ts 响应体样本。每个接口的 mock 数据就是前端期望的真实返回结构。实现完接口直接 diff 一下就知道字段对没对齐
web/src/api/client.ts HTTP 客户端封装。看一下能理解前端对响应体、鉴权头、错误的处理逻辑
web/src/api/*.ts (除 _mock 外) 各资源的接口调用代码,显示了实际请求的 URL、HTTP 方法、参数形状
web/dist/ 已构建好的前端产物,可直接部署联调

2. 必须遵守的 3 条全局约定

2.1 统一响应信封

所有 JSON 接口(包括成功和业务失败)都必须返回:

{
  "code": 0,            // 业务码: 0 = 成功,非 0 = 业务错误
  "message": "ok",      // 错误信息(成功时随意,业务错误时给用户能看懂的中文)
  "data": { ... },      // 实际数据;失败时可以为 null 或 { errors: [...] }
  "request_id": "..."   // 可选,排查日志用
}
不要直接返回数据对象。前端拦截器会读 code,如果不是 0 就 throw 出去,UI 显示报错。所以业务失败也要 HTTP 200 + code != 0,而不是 HTTP 4xx。

2.2 鉴权

  • 请求头 Authorization: Bearer <token>,前端自动带
  • HTTP 401 → 前端自动登出并跳 /login。所以 token 过期/无效用 401
  • 响应头里如果有 X-Refresh-Token: <new_token>,前端会写回 store → 用这个头做无感续签

2.3 分页

列表接口统一形如:

{
  "list": [...],        // 当前页数据
  "page": 1,
  "page_size": 12,
  "total": 47
}

4. 联调起步流程

  1. P0 接口先实现:GET /projects + GET /projects/:id + GET /characters + POST /auth/login。这四个跑通,工作台、编辑器、角色库、登录就都能联调
  2. 本地启动:后端起在 http://localhost:8080(改端口的话改 web/vite.config.ts 里的 proxy target)
  3. 前端切真实模式:在 web/.env.development 改 VITE_USE_MOCK=false,然后 cd web && npm run dev
  4. 访问 http://localhost:5173,DevTools Network 应该能看到对 /api/v1/* 的真实请求(Vite 代理到后端 8080)

部署联调(同源方案,推荐)

把 web/dist/ 拷到后端服务器,用 nginx 或后端框架的静态目录托管,把 /api/v1/* 转给后端进程,其他路径 fallback 到 index.html。最简 nginx 示例:

server {
  listen 80;
  root /var/www/duanju/dist;

  # API 走后端
  location /api/ {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  # 静态资源
  location /assets/ {
    try_files $uri =404;
    expires 1y;
    add_header Cache-Control "public, immutable";
  }

  # SPA fallback
  location / {
    try_files $uri $uri/ /index.html;
  }
}
跨域方案(前后端不同源)也行,但要后端开 CORS 并在前端 build 时设 VITE_API_BASE_URL=https://api.xxx.com/v1。同源更省事,推荐同源。

5. 验收信号

联调时按这个顺序自测:

  1. 访问 /dashboard:看到项目卡片网格,状态筛选段控件数字正确
  2. 点任意项目卡片 → 进入编辑器:左侧 9 个全局字段 + N 个分镜字段全部加载出原值,角色列表正确
  3. /characters:能新建/编辑/删除角色,刷新后保留
  4. /login:手机验证码登录返回 token 后跳回首页,刷新页面仍登录(token 持久化到 localStorage)

6. 联系 / 待对齐项

  • 契约里 §5 待对齐 还有几条没敲定(SSE 心跳格式、图片上传分片阈值等),实现时遇到再讨论
  • 字段命名前后端用 snake_case(如 shot_count、cover_url),前端 TS 里也是这么写的,不要 camelCase
  • 有任何字段不确定的地方,优先看 web/src/api/_mock.ts 里的样本数据 — 那个就是前端实际期望拿到的形状
